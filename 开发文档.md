# SQL 编辑器项目详细开发文档
## 一、项目概述
### 1.1 项目背景与目标
在企业研发、数据分析、数据库运维等场景中，开发人员经常需要快速连接数据库、编写并执行 SQL 语句进行数据查询与调试。传统客户端工具（如 Navicat、DBeaver）需要本地安装、配置复杂，且无法便捷地进行团队共享与权限管控。

本项目旨在打造一款 **Web 端在线 SQL 编辑器**，采用前后端分离架构，实现数据库连接可视化管理、SQL 在线编辑执行、结果表格化展示、多层安全防护等核心能力，无需安装客户端，通过浏览器即可完成全流程数据库操作，兼顾易用性、安全性与可扩展性。

### 1.2 核心功能清单
| 功能模块 | 详细能力 |
|----------|----------|
| 数据源管理 | 可视化新增数据库连接、连接连通性测试、连接列表查询、连接删除、多连接一键切换 |
| SQL 编辑能力 | MySQL 语法高亮、代码自动缩进、括号匹配、行号显示、多光标编辑、SQL 一键格式化 |
| SQL 执行能力 | 全量执行、选中片段执行、执行耗时统计、多语句支持、执行中途可终止 |
| 结果展示 | 查询结果表格化渲染、列宽自适应、长文本省略、滚动浏览、影响行数展示 |
| 安全防护 | 高危语句拦截、只读模式管控、无 WHERE 写操作拦截、查询行数默认限制、连接权限隔离 |
| 辅助能力 | Ctrl+Enter 快捷键执行、编辑器清空、执行状态提示、错误信息友好展示 |

### 1.3 适用业务场景
- 企业内部数据查询与分析平台
- 开发/测试环境快速 SQL 调试工具
- 数据库教学实训与在线实验平台
- 数据中台配套 SQL 执行组件
- 运维人员数据库日常运维工具

## 二、整体技术架构
### 2.1 系统架构图与数据流
采用经典三层前后端分离架构，数据流全程可追溯：
```
用户浏览器
    ↓
前端 React 应用（编辑器交互、结果渲染、连接管理）
    ↓ HTTP/HTTPS
后端 Spring Boot 应用
    ├─ 控制层：接口接收、参数校验、跨域处理
    ├─ 业务层：安全校验、数据源路由、SQL 执行、结果封装
    └─ 数据层：动态连接池管理、JDBC 执行 SQL
        ↓
目标数据库（MySQL/PostgreSQL 等）
```

**核心执行流程**：
1. 用户在前端选择数据源，编写 SQL 语句，点击执行
2. 前端将 SQL 语句 + 数据源 ID 发送至后端接口
3. 后端接收请求后，根据数据源 ID 切换到对应数据库连接
4. 执行安全校验规则，拦截高危与违规语句
5. 通过 JDBC 执行 SQL，获取执行结果
6. 结果标准化封装后返回前端
7. 前端解析结果，以表格形式展示查询数据或执行状态

### 4.3 核心模块实现
#### 4.3.1 动态数据源模块
##### 4.3.1.1 设计原理
基于 Spring 提供的 `AbstractRoutingDataSource` 抽象类实现动态数据源路由。其核心逻辑是：在执行数据库操作前，通过 `determineCurrentLookupKey()` 方法获取当前要使用的数据源标识，然后从目标数据源集合中找到对应的数据源进行使用。

配合 `ThreadLocal` 在线程内存储数据源标识，保证同一线程内的数据库操作使用同一个连接，不同线程之间互不干扰，执行完成后清理线程变量，避免线程复用导致数据源错乱。

##### 4.3.1.2 数据源上下文持有器
```java
package com.example.sqleditor.datasource;

/**
 * 动态数据源上下文持有器
 * 使用ThreadLocal存储当前线程使用的数据源key，保证线程安全
 */
public class DynamicDataSourceContextHolder {

    // ThreadLocal存储数据源key，每个线程独立拥有一份副本
    private static final ThreadLocal<String> CONTEXT_HOLDER = new ThreadLocal<>();

    /**
     * 设置当前线程使用的数据源key
     * @param key 数据源标识
     */
    public static void setDataSourceKey(String key) {
        CONTEXT_HOLDER.set(key);
    }

    /**
     * 获取当前线程使用的数据源key
     * @return 数据源标识
     */
    public static String getDataSourceKey() {
        return CONTEXT_HOLDER.get();
    }

    /**
     * 清除当前线程的数据源key
     * 必须在请求结束后调用，防止线程复用导致数据错乱
     */
    public static void clearDataSourceKey() {
        CONTEXT_HOLDER.remove();
    }
}
```

**设计要点**：
- 使用 `ThreadLocal` 而非普通变量，确保多并发场景下数据源切换不互相影响
- 提供 `clear` 方法，配合 `finally` 代码块使用，避免内存泄漏与连接串扰
- 无状态设计，全部为静态方法，方便全局调用

##### 4.3.1.3 动态数据源路由类
```java
package com.example.sqleditor.datasource;

import org.springframework.jdbc.datasource.lookup.AbstractRoutingDataSource;

/**
 * 动态数据源路由类
 * 继承AbstractRoutingDataSource，重写数据源查找逻辑
 */
public class DynamicDataSource extends AbstractRoutingDataSource {

    /**
     * 确定当前使用的数据源key
     * Spring在获取连接前会调用此方法，根据返回值选择对应的数据源
     * @return 数据源标识
     */
    @Override
    protected Object determineCurrentLookupKey() {
        return DynamicDataSourceContextHolder.getDataSourceKey();
    }
}
```

**工作机制**：
1. Spring 初始化时，将所有目标数据源存入 `targetDataSources` Map 中
2. 每次执行数据库操作获取连接时，调用 `determineCurrentLookupKey()` 获取 key
3. 根据 key 从 Map 中查找对应的数据源，进而获取数据库连接

##### 4.3.1.4 数据源配置类
```java
package com.example.sqleditor.config;

import com.example.sqleditor.datasource.DynamicDataSource;
import com.zaxxer.hikari.HikariDataSource;
import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.Primary;
import org.springframework.jdbc.core.JdbcTemplate;

import javax.sql.DataSource;
import java.util.HashMap;
import java.util.Map;

/**
 * 动态数据源配置类
 * 负责初始化默认数据源、动态数据源、JdbcTemplate
 */
@Configuration
public class DynamicDataSourceConfig {

    /**
     * 创建默认数据源，读取application.yml中的spring.datasource配置
     * 用于存储系统本身的数据源配置数据
     */
    @Bean
    @ConfigurationProperties(prefix = "spring.datasource")
    public DataSource defaultDataSource() {
        return new HikariDataSource();
    }

    /**
     * 创建动态数据源，作为系统主数据源
     * @Primary 标注为首选数据源，当有多个DataSource实例时优先使用此实例
     */
    @Bean
    @Primary
    public DynamicDataSource dynamicDataSource(DataSource defaultDataSource) {
        // 目标数据源集合，key为数据源标识，value为数据源实例
        Map<Object, Object> targetDataSources = new HashMap<>();
        targetDataSources.put("default", defaultDataSource);

        DynamicDataSource dynamicDataSource = new DynamicDataSource();
        // 设置所有目标数据源
        dynamicDataSource.setTargetDataSources(targetDataSources);
        // 设置默认数据源，当未指定key时使用
        dynamicDataSource.setDefaultTargetDataSource(defaultDataSource);
        return dynamicDataSource;
    }

    /**
     * 配置JdbcTemplate，使用动态数据源
     * 所有通过JdbcTemplate执行的SQL都会自动走动态数据源路由
     */
    @Bean
    public JdbcTemplate jdbcTemplate(DynamicDataSource dynamicDataSource) {
        return new JdbcTemplate(dynamicDataSource);
    }
}
```

#### 4.3.2 数据源管理模块
##### 4.3.2.1 连接测试实现
新增数据源前，先验证连接配置是否正确，避免无效配置入库。原理是临时创建一个 Hikari 数据源，尝试获取数据库连接，获取成功则配置有效，获取失败则配置错误。

```java
/**
 * 测试数据库连接是否可用
 * @param request 数据源配置信息
 * @return true-连接成功 false-连接失败
 */
public boolean testConnection(DatasourceAddRequest request) {
    HikariDataSource tempDataSource = null;
    try {
        // 构建临时数据源
        tempDataSource = buildHikariDataSource(request);
        // 尝试获取连接，超时时间2秒
        try (Connection connection = tempDataSource.getConnection()) {
            return connection.isValid(2);
        }
    } catch (Exception e) {
        // 连接异常，返回失败
        return false;
    } finally {
        // 关闭临时数据源，释放资源
        if (tempDataSource != null) {
            tempDataSource.close();
        }
    }
}
```

##### 4.3.2.2 数据源动态加载
用户新增数据源后，需要将其动态添加到动态数据源的目标集合中，无需重启项目即可使用。

```java
/**
 * 将数据源加载到动态上下文
 * @param key 数据源标识（使用数据库主键ID）
 * @param request 数据源配置
 */
private void loadDataSourceToContext(String key, DatasourceAddRequest request) {
    // 1. 创建Hikari连接池实例
    HikariDataSource dataSource = buildHikariDataSource(request);

    // 2. 获取当前已有的所有数据源，追加新数据源
    Map<Object, Object> targetDataSources = new HashMap<>(
            dynamicDataSource.getResolvedDataSources()
    );
    targetDataSources.put(key, dataSource);

    // 3. 重新设置目标数据源并刷新
    dynamicDataSource.setTargetDataSources(targetDataSources);
    dynamicDataSource.afterPropertiesSet();
}

/**
 * 构建Hikari数据源实例
 * @param request 配置参数
 * @return Hikari数据源
 */
private HikariDataSource buildHikariDataSource(DatasourceAddRequest request) {
    HikariConfig config = new HikariConfig();
    config.setJdbcUrl(request.getUrl());
    config.setUsername(request.getUsername());
    config.setPassword(request.getPassword());

    // 根据数据库类型选择驱动
    String driverClass = "mysql".equals(request.getDbType())
            ? "com.mysql.cj.jdbc.Driver"
            : "org.postgresql.Driver";
    config.setDriverClassName(driverClass);

    // 连接池参数配置
    config.setMaximumPoolSize(5);      // 每个数据源最大5个连接
    config.setMinimumIdle(1);          // 最小1个空闲连接
    config.setConnectionTimeout(3000); // 连接超时3秒
    config.setIdleTimeout(60000);      // 空闲连接60秒后释放
    config.setMaxLifetime(1800000);    // 连接最大生命周期30分钟

    return new HikariDataSource(config);
}
```

##### 4.3.2.4 数据源删除逻辑
删除数据源时，不仅要删除数据库中的配置记录，还要关闭连接池并从动态上下文中移除，避免连接泄漏。

```java
/**
 * 删除数据源
 * @param id 数据源ID
 */
public void delete(Long id) {
    // 1. 从数据库删除配置记录
    jdbcTemplate.update("DELETE FROM datasource_config WHERE id = ?", id);

    // 2. 从动态数据源中移除并关闭连接池
    String key = id.toString();
    Map<Object, Object> targetDataSources = dynamicDataSource.getResolvedDataSources();
    
    if (targetDataSources.containsKey(key)) {
        DataSource dataSource = (DataSource) targetDataSources.get(key);
        // 关闭连接池，释放所有连接
        if (dataSource instanceof HikariDataSource hikariDs) {
            hikariDs.close();
        }
        targetDataSources.remove(key);
        
        // 刷新动态数据源
        dynamicDataSource.setTargetDataSources(targetDataSources);
        dynamicDataSource.afterPropertiesSet();
    }
}
```

#### 4.3.3 SQL执行引擎模块
##### 4.3.3.1 执行流程
SQL 执行遵循严格的流水线处理，每一步校验不通过则直接返回：
1. 切换数据源：根据传入的数据源 ID，设置线程上下文
2. 基础校验：SQL 非空校验、去除末尾分号
3. 安全校验：高危语句、只读模式、全表操作拦截
4. 类型判断：区分查询类语句与写操作类语句
5. 执行 SQL：调用 JdbcTemplate 对应方法执行
6. 结果封装：统一格式封装执行结果
7. 清理上下文：finally 块中清理线程变量

##### 4.3.3.2 安全校验机制
四层安全防护，从外到内保障数据库安全：
```java
/**
 * SQL安全校验
 * @param sql 待校验的SQL语句
 * @return null-校验通过，非null-校验失败，返回错误信息
 */
private String securityCheck(String sql) {
    String upperSql = sql.toUpperCase();

    // 第一层：高危破坏性语句强制拦截
    if (upperSql.contains("DROP ") || upperSql.contains("TRUNCATE ")) {
        return "禁止执行 DROP / TRUNCATE 等高危破坏性语句";
    }

    // 第二层：只读模式下拦截所有写操作
    if (readOnly) {
        if (upperSql.startsWith("INSERT") || upperSql.startsWith("UPDATE")
                || upperSql.startsWith("DELETE") || upperSql.startsWith("ALTER")
                || upperSql.startsWith("CREATE") || upperSql.startsWith("RENAME")) {
            return "当前为只读模式，禁止执行写操作语句";
        }
    }

    // 第三层：拦截不带WHERE条件的DELETE和UPDATE，防止全表误删改
    if ((upperSql.startsWith("DELETE") || upperSql.startsWith("UPDATE"))
            && !upperSql.contains("WHERE")) {
        return "禁止执行不带 WHERE 条件的 DELETE / UPDATE 语句";
    }

    // 第四层：SQL长度限制，防止超长SQL攻击
    if (sql.length() > 10000) {
        return "SQL语句长度超出限制";
    }

    return null;
}
```

##### 4.3.3.3 语句类型分发
根据 SQL 前缀判断语句类型，选择不同的执行方法：
- 查询类（SELECT、SHOW、DESC、EXPLAIN）：调用 `queryForList`，返回结果集
- 增删改类（INSERT、UPDATE、DELETE、ALTER 等）：调用 `update`，返回影响行数

```java
String sqlUpper = sql.toUpperCase().trim();
if (sqlUpper.startsWith("SELECT") || sqlUpper.startsWith("SHOW")
        || sqlUpper.startsWith("DESC") || sqlUpper.startsWith("EXPLAIN")) {
    // 查询类语句：自动追加LIMIT保护
    String finalSql = appendLimitIfNeed(sql);
    List<Map<String, Object>> rows = jdbcTemplate.queryForList(finalSql);
    List<String> columns = rows.isEmpty() ? List.of() : new ArrayList<>(rows.get(0).keySet());
    
    builder.success(true)
            .columns(columns)
            .rows(rows)
            .message("查询成功，共 " + rows.size() + " 条");
} else {
    // 增删改类语句
    int affectedRows = jdbcTemplate.update(sql);
    builder.success(true)
            .affectedRows(affectedRows)
            .message("执行成功，影响 " + affectedRows + " 行");
}
```

##### 4.3.3.4 查询行数保护
对查询语句自动追加 LIMIT 限制，避免用户忘记加 LIMIT 导致全表扫描，拖垮数据库性能。
```java
/**
 * 为SELECT语句自动追加LIMIT限制
 * @param sql 原始SQL
 * @return 追加LIMIT后的SQL
 */
private String appendLimitIfNeed(String sql) {
    if (sql.toUpperCase().contains("LIMIT")) {
        // 已经包含LIMIT则不处理
        return sql;
    }
    return sql + " LIMIT " + defaultLimit;
}
```

## 七、部署与上线
### 7.1 开发环境部署
开发环境采用前后端分离启动，前端通过代理访问后端，调试便捷：
1. 后端：IDE 直接启动，端口 8080
2. 前端：`npm run dev` 启动，端口 5173
3. 访问地址：`http://localhost:5173`

### 7.2 生产环境部署
#### 7.2.1 后端打包与部署
1. **打包**：在项目根目录执行
   ```bash
   mvn clean package -DskipTests
   ```
   打包完成后，`target` 目录下生成 `sql-editor-backend.jar`

2. **上传部署**：将 jar 包上传至服务器，使用以下命令后台启动
   ```bash
   nohup java -jar sql-editor-backend.jar --spring.profiles.active=prod > app.log 2>&1 &
   ```

3. **生产环境配置建议**：
   - 单独创建 `application-prod.yml`，配置生产环境数据库信息
   - 开启只读模式：`sql.read-only: true`
   - 调大连接池参数，适配并发访问
   - 使用独立的数据库只读账号，最小权限原则

#### 7.2.2 前端打包与Nginx部署
1. **打包**：执行
   ```bash
   npm run build
   ```
   打包完成后生成 `dist` 目录

2. **Nginx 配置**：
   ```nginx
   server {
       listen 80;
       server_name sql-editor.example.com; # 替换为你的域名

       # 前端静态资源
       location / {
           root /data/sql-editor/dist;
           index index.html;
           try_files $uri $uri/ /index.html; # SPA路由兼容
       }

       # 后端接口反向代理
       location /api/ {
           proxy_pass http://127.0.0.1:8080/api/;
           proxy_set_header Host $host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
           proxy_connect_timeout 30s;
           proxy_read_timeout 30s;
       }

       # 静态资源缓存优化
       location ~* \.(js|css|png|jpg|gif)$ {
           expires 7d;
       }
   }
   ```

3. **重载 Nginx** 使配置生效，访问域名即可使用

## 八、安全设计规范
### 8.1 数据库权限管控
- **最小权限原则**：生产环境连接数据库的账号仅授予必要权限，只读场景仅给 SELECT 权限
- **账号隔离**：系统默认数据源与业务数据源使用不同账号，避免权限交叉
- **禁止超级账号**：严禁使用 root、super 权限账号连接编辑器
- **敏感表隔离**：包含敏感数据的表建议通过视图做权限隔离，禁止直接访问

### 8.2 SQL注入防护
- 本系统直接执行用户输入的 SQL，天然存在 SQL 注入风险，因此必须通过权限管控限制危害范围
- 生产环境强制使用只读账号，即使注入也只能查询数据，无法篡改或删除
- 禁止将编辑器直接暴露到公网，仅限内网访问
- 对接企业登录体系，实名访问，所有操作可追溯

### 8.3 高危操作拦截机制
四层拦截体系，从配置到代码层层防护：
1. **数据库账号层**：只读账号无法执行写操作
2. **系统配置层**：全局只读开关，一键关闭所有写操作
3. **代码逻辑层**：DROP、TRUNCATE 等高危语句强制拦截
4. **操作规范层**：无 WHERE 的 DELETE/UPDATE 强制拦截

### 8.4 密码安全设计
- 数据库中存储的数据库密码，生产环境建议使用 AES 对称加密存储，加载时解密
- 列表接口不返回密码字段，前端无法获取明文密码
- 密码传输建议使用 HTTPS 加密，避免明文传输
- 定期轮换数据库密码，降低泄露风险

### 8.5 操作审计与追溯
- 所有 SQL 执行必须记录日志，包含：操作人、IP、时间、数据源、SQL 内容、执行结果、耗时
- 新增、删除数据源操作记录审计日志
- 日志保留至少 6 个月，支持事后审计排查
- 对接企业统一登录，实现实名操作

### 8.6 跨域安全配置
- 开发环境使用通配符跨域，生产环境必须关闭 `@CrossOrigin` 通配符
- 生产环境通过 Nginx 反向代理统一入口，前后端同域名，避免跨域
- 如需开放跨域，仅配置指定可信域名，禁止使用 `*` 通配符

## 十、常见问题与排查
### 10.1 后端启动类问题
**问题1：启动报错 `Cannot load driver class: com.mysql.cj.jdbc.Driver`**
- 原因：MySQL 驱动依赖未正确引入
- 解决：检查 pom.xml 中是否添加 mysql-connector-j 依赖，刷新 Maven 依赖

**问题2：启动报错 `Access denied for user 'root'@'localhost'`**
- 原因：数据库账号密码配置错误
- 解决：检查 application.yml 中的 username 和 password 是否正确

**问题3：动态数据源切换不生效，一直用默认数据源**
- 原因：ThreadLocal 未正确设置，或 JdbcTemplate 未使用动态数据源
- 解决：
  1. 确认执行 SQL 前调用了 `DynamicDataSourceContextHolder.set()`
  2. 确认 JdbcTemplate 的 Bean 注入的是 DynamicDataSource
  3. 确认 finally 块中未提前清理上下文

### 10.2 前端运行类问题
**问题1：编辑器区域空白，不显示**
- 原因：Monaco Editor worker 文件加载失败
- 解决：Vite 项目可安装 `vite-plugin-monaco-editor` 插件处理 worker 打包

**问题2：接口请求报跨域错误**
- 原因：开发环境代理配置错误，或直接请求后端地址
- 解决：
  1. 确认 vite.config.js 中 proxy 配置正确
  2. 前端请求使用相对路径 `/api/xxx`，不要写完整 IP

**问题3：格式化功能无反应**
- 原因：SQL 语法错误，格式化抛出异常被捕获
- 解决：检查 SQL 语法，修正括号、关键字等错误后重试

### 10.3 数据库连接类问题
**问题1：测试连接一直失败**
- 排查步骤：
  1. 确认数据库服务已启动
  2. 确认 IP、端口、库名正确
  3. 确认账号密码正确
  4. 确认数据库允许当前服务器远程访问
  5. 检查防火墙与安全组是否开放端口

**问题2：连接一段时间后失效**
- 原因：连接空闲超时被数据库断开
- 解决：HikariCP 配置连接测试，设置 `connection-test-query: SELECT 1`

### 10.4 功能异常类问题
**问题1：SELECT 查询没有返回全部数据**
- 原因：默认追加了 LIMIT 限制
- 解决：SQL 中手动指定更大的 LIMIT，或修改配置文件中的 `default-limit` 参数

**问题2：多语句执行只返回第一条结果**
- 原因：当前实现默认只处理单条语句
- 解决：可扩展实现按分号拆分语句，逐条执行，合并返回结果
